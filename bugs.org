** Root Cause
:PROPERTIES:
:CUSTOM_ID: root-cause-2
:END:
=HorizontalAlignment="Center"= / =VerticalAlignment="Center"= (or
default =Stretch= with explicit Width/Height) causes the Image to be
centered within the ScrollViewer's content area. The scroll range
assumes content origin at (0,0), but centering shifts the image so that
right/bottom edges extend past the scrollable range. Purely removing
both attributes reverts to default =Stretch=, which ALSO centers
explicit-sized elements --- same clipping.

** Fix
:PROPERTIES:
:CUSTOM_ID: fix-3
:END:
Set =HorizontalAlignment="Left"= and =VerticalAlignment="Top"= so the
Image sits at (0,0) in the ScrollViewer content area. The scroll range
[0, extent − viewport] correctly covers the full image. Visual centering
is handled by scroll offset calculation in
=RecenterScrollViewerAfterLayout()=.

** Sub-bug A: preserve-center formula is viewport-agnostic
:PROPERTIES:
:CUSTOM_ID: sub-bug-a-preserve-center-formula-is-viewport-agnostic
:END:
*** Root Cause
:PROPERTIES:
:CUSTOM_ID: root-cause-3
:END:
The preserve-center formula
=newOffset = oldOffset + (newExtent − oldExtent)/2= only gives correct
results when =oldOffset = (extent − viewport)/2= (image perfectly
centered). For a panned image the formula fails because it ignores
viewport size. Example: scrolled to Y offset 50 in a 1000×1000 extent
with 600px viewport --- old formula yields Y=300 on 1.5× zoom instead of
correct Y=225 (75px jump).

*** Fix
:PROPERTIES:
:CUSTOM_ID: fix-4
:END:
Replaced with =newOffset = (oldOffset + vp/2) * zoomRatio − vp/2= which
properly factors in viewport: the viewport-center point in content
coords is =oldOffset + vp/2=, scaled by zoom ratio, then back to offset.

* B36: =o= key doesn't fire from WebView (tunneling vs bubbling)
:PROPERTIES:
:CUSTOM_ID: b36-o-key-doesnt-fire-from-webview-tunneling-vs-bubbling
:END:
** Location
:PROPERTIES:
:CUSTOM_ID: location-14
:END:
- =Views/MainWindow.axaml.cs= --- =OnKeyDown()=, =HandleKey()=,
  tunneling handler

** Status
:PROPERTIES:
:CUSTOM_ID: status-21
:END:
- OPEN --- needs a clean solution

** Root Cause
:PROPERTIES:
:CUSTOM_ID: root-cause-12
:END:
=KeyDown= is a bubbling routed event. When the NativeWebView has focus
and a plain letter key (=o=) is pressed, the WebView consumes the event
for text input and never bubbles it to the MainWindow. =?= (Shift+Oem2)
and =Ctrl+,= (Ctrl+OemComma) work because the WebView doesn't treat
modifier-combination keys as text input.

The tunneling workaround
(=AddHandler(KeyDownEvent, handler, RoutingStrategies.Tunnel)=) catches
=o= before the WebView gets it, but requires an =e.Source is TextBox=
guard to not steal input from real text boxes. This is fragile.

** Proper fix needed
:PROPERTIES:
:CUSTOM_ID: proper-fix-needed
:END:
NativeWebView should either:

1. Not consume plain letter keys in read-only mode (it's a display-only
   WebView)
2. Forward unhandled key events to the parent window

** Workaround applied
:PROPERTIES:
:CUSTOM_ID: workaround-applied
:END:
Tunneling handler in =MainWindow()= constructor, guarded by
=e.Source is TextBox=.

*** Summary of what's happening
:PROPERTIES:
:CUSTOM_ID: summary-of-whats-happening
:END:
The *link-opening pipeline* has two stages:

1. *=HtmlContentBuilder.cs=* --- renders all links as
   =<a data-href="url">= (no real =href=) and a JS click handler
   forwards them to the host app via an
   =http://openurl.local/<encoded-url>= iframe navigation.
2. *=MainWindow.axaml.cs=* --- =OnWebViewNavigationStarted= catches
   =openurl.local= and calls =OpenInBrowser(url)=.

The crash lives entirely in =OpenInBrowser='s Linux path.

--------------

*** Analysis of =OpenInBrowser= (Linux section)
:PROPERTIES:
:CUSTOM_ID: analysis-of-openinbrowser-linux-section
:END:
#+begin_src csharp
var chromiumBrowsers = new[] { "brave", "google-chrome", "chromium-browser", "chromium" };
foreach (var browser in chromiumBrowsers)
{
    try { Process.Start(browser, $"--ozone-platform-hint=x11 {url}"); return; }
    catch { }
}
try { Process.Start("firefox", url); return; }
catch { }
// Last resort: xdg-open
try { Process.Start(new ProcessStartInfo { FileName = url, UseShellExecute = true }); }
catch (Exception ex) { Log.Warning(...); }
#+end_src

1. Problems with the current approach

   *1. The loop never skips browsers that aren't installed.*
   =Process.Start("brave", ...)= does /not/ throw when =brave= doesn't
   exist on the =PATH= --- on Linux with the default =ProcessStartInfo=,
   it may silently launch a shell command that fails at the OS level
   rather than throwing back to C#. The =catch {}= only catches
   exceptions thrown by =Process.Start= itself (e.g. =Win32Exception= if
   the executable isn't found). The result: on systems where =brave=
   /is/ present but crashes, execution returns immediately before any
   fallback is tried.

   *Fix*: check if the binary exists first (=which brave= /
   =File.Exists(which result)=) or use =ProcessStartInfo= with
   =UseShellExecute = false= and catch =Win32Exception= specifically ---
   that exception /is/ reliably thrown when the binary is not on =PATH=.

   *2. The URL is passed as part of the argument string --- unsafe.*
   =$"--ozone-platform-hint=x11 {url}"= puts the raw URL into a single
   argument string. If the URL contains spaces or shell-special
   characters this will either break or (on =UseShellExecute = true=) be
   interpreted by the shell. Use =ArgumentList= instead:

   #+begin_src csharp
   new ProcessStartInfo(browser)
   {
       UseShellExecute = false,
       ArgumentList = { "--ozone-platform-hint=x11", url }
   }
   #+end_src

   *3. No =WaitForExit= / process supervision --- cannot detect crash.*
   Brave's segfault happens in the child process, not in the
   =Process.Start= call itself. The current code returns immediately
   after starting the process, so there is no way to detect that Brave
   crashed and fall through to the next candidate.

   *4. The =xdg-open= last resort uses =UseShellExecute = true= with the
   URL as =FileName=.* This works on Windows (Shell interprets a URL as
   "open in browser"), but on Linux =UseShellExecute = true= delegates
   to the =$BROWSER= env var / =xdg-open= only if the OS desktop
   integration handles it. The correct last resort on Linux is
   =Process.Start("xdg-open", url)=.

   *5. =firefox= does not need =--ozone-platform-hint=x11=*, as the org
   file correctly notes, but that's already done correctly --- it's
   tried separately without the flag.

   --------------

*** Recommended fix for =OpenInBrowser= (Linux)
:PROPERTIES:
:CUSTOM_ID: recommended-fix-for-openinbrowser-linux
:END:
#+begin_src csharp
// Linux: Chromium-family browsers crash on Wayland; force XWayland via --ozone-platform-hint=x11.
// Firefox runs fine without it.

if (TryStartBrowser("firefox", url))       // Firefox first — no flag needed, no Wayland crash
    return;

var chromiumBrowsers = new[] { "brave", "google-chrome", "chromium-browser", "chromium" };
foreach (var browser in chromiumBrowsers)
{
    if (TryStartBrowser(browser, url, "--ozone-platform-hint=x11"))
        return;
}

// Last resort: xdg-open
try
{
    Process.Start("xdg-open", url);
}
catch (Exception ex)
{
    Log.Warning(ex, "Failed to open URL via xdg-open: {Url}", url);
}

// ─────────────────────────────────────────
static bool TryStartBrowser(string binary, string url, string? extraArg = null)
{
    try
    {
        var psi = new ProcessStartInfo(binary) { UseShellExecute = false };
        if (extraArg != null) psi.ArgumentList.Add(extraArg);
        psi.ArgumentList.Add(url);
        Process.Start(psi);
        return true;
    }
    catch (System.ComponentModel.Win32Exception)
    {
        // Binary not found on PATH — try next
        return false;
    }
    catch (Exception ex)
    {
        Log.Warning(ex, "Failed to launch browser {Binary}", binary);
        return false;
    }
}
#+end_src

Key changes: - =UseShellExecute = false= + catching =Win32Exception=
reliably skips missing binaries. - =ArgumentList= avoids
argument-injection/splitting bugs. - Firefox is tried *first* since it
doesn't need the flag and is widely installed. - =xdg-open= as last
resort uses the correct invocation.
